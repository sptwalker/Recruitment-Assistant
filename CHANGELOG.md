# V3.24 版本更新说明

发布日期：2026-07-21

本版汇总 V3.13–V3.24 的迭代，核心是**多接口 AI 降级、岗位匹配提效、A 组加固与 M1 单库化架构整改**。

## 架构：统一到单一 SQLite 库（M1 地基整改）

- **移除捆绑 PostgreSQL**：原本岗位/采集在 PG、候选人在 SQLite 的双库结构，统一并入单一 SQLite（`data/resume_archive.db`）。安装包体积 122MB → **98MB**。
- 启动器不再编排 PostgreSQL，删除中文路径的 NTFS junction 黑魔法、trust 鉴权、硬编码密码。
- `PositionMatch.position_id` 改为**真外键**指向同库 `job_position`，`save_position_match` 去掉裸 sqlite3 绕行、改 ORM upsert。
- 删除死代码（重复的 PG candidate/resume/export/audit 家族及其死服务）。
- **老版升级自动迁移**：首次启动检测到旧 pgdata 时，一次性把 5 张活表从旧 PG 迁入 SQLite——迁移前自动备份、幂等 marker、坏行跳过、失败可重试、不删旧数据。
- alembic 重建为单一 SQLite metadata 的迁移源。

## AI 接口：多接口 + 自动降级

- 「系统设置 → AI模型」支持**多个 OpenAI 兼容接口**卡片式增删（名称/地址/Key/模型）。
- 单选**主接口**用于所有调用，其余启用接口作**备用**；主接口失败**按调用自动降级**（不粘死，主接口恢复后自动切回）并提示用户；降级通知按会话隔离。
- 可为**简历解析**单独指定更快的模型（解析不必用推理大模型）。
- 端点不支持 `response_format=json_object` 时**自动去参重试**，兼容 GLM/Claude 代理等更多端点。

## 岗位匹配优化

- **JD 哈希复用**：只对「未评分」或「JD 已变化」的候选人调用 LLM，其余复用库中评分，避免重复扣费；另加「🔄 重新评分」按钮做强制全量重算。
- **实时进度 + 停止**：智能匹配改为逐批 rerun 状态机——真实进度条显示「已评估 xx/xx 份简历」，匹配中可随时「⏹ 停止匹配」；**每批即时保存**，停止/中断不丢已评结果。

## A 组加固

- **数据备份**：系统设置内一键备份简历库 + 导出候选人 Excel（防误删，PII 保护）。
- **测试 + CI**：新增去重/匹配复用/降级/备份/迁移等核心链路测试（共 98 项），配 GitHub Actions 自动跑。
- 启动迁移失败不再静默吞掉，改为 loguru 记录。
- **安全**：`data/ai_models.json`（含真实 Key）移出 git 跟踪，改用 `.example` 模板；请自行轮换曾入库的 Key。

## 采集健壮性（BOSS直聘）

- 对方撤回简历、「同意」按钮不可点击时果断跳过候选人；附件下载最多重试 3 次后强行跳过，避免采集卡死。
- 启动器改按端口复用/停止 Streamlit，修复 WebSocket 10048 端口占用。

# V3.12 版本更新说明

发布日期：2026-06-24

## 核心修复

### 修复按钮不可点击问题

**问题**：V3.11 安装后"匹配所有岗位"和"智能匹配"按钮变成不可点击状态

**修复**：自动匹配流程添加 try/except，异常时自动恢复按钮状态

### 修复匹配结果全部保存失败

**问题**：日志分析发现所有 420 条匹配结果都因 `FOREIGN KEY constraint failed` 无法保存，
但数据库中确实有 499 个候选人（ID 2-500）。说明 FK 约束本身有问题，而非数据缺失。

**根本原因**：
`position_matches` 表的 `candidate_id` 外键是在 `PRAGMA foreign_keys=ON` 下被检查的，
而 `position_id` 引用的是 PostgreSQL 的 `job_position`（跨库），SQLite 无法验证这个跨库引用，
ORM 的 `session.merge()` 生成的 `INSERT ... RETURNING` 语句触发了 SQLite 的 FK 完整性检查。

**解决方案**：
`save_position_match()` 改用**原始 sqlite3 连接**直接执行 `INSERT OR REPLACE`：
- 绕过 SQLAlchemy 的 FK 强制检查（`PRAGMA foreign_keys=ON` 是 SQLAlchemy 连接事件设置的）
- 原始 sqlite3 连接默认 `PRAGMA foreign_keys=OFF`，不受 ORM 层面的 FK 约束影响
- candidate_id 的有效性已在匹配开始前通过数据验证（直接 SQL 查询）确认
- 使用 `INSERT OR REPLACE` 自动处理重复的 (position_id, candidate_id) 组合

**效果**：
- ✅ 彻底解决保存失败问题
- ✅ 不再依赖 SQLAlchemy session 的 FK 状态
- ✅ 性能更好（无 ORM 开销）
- ✅ 调试日志包含 FK 诊断信息用于后续排查

# V3.11 版本更新说明

发布日期：2026-06-23

## 核心修复

### 修复保存时的候选人验证逻辑

**问题诊断**：
通过调试日志发现：
- 数据验证阶段：查询到499个候选人（正确）
- 保存阶段：360条全部失败（FOREIGN KEY constraint）
- 说明保存时的验证使用了不同的数据源

**根本原因**：
`save_position_match()` 中使用ORM查询 `select(func.count())` 验证候选人存在性，但ORM可能返回缓存的结果，与数据库实际状态不一致。

**解决方案**：
保存时改用**直接SQL查询**验证候选人是否存在：
```python
# 旧方案（可能使用缓存）
candidate_exists = self.session.scalar(
    select(func.count()).where(Candidate.candidate_id == candidate_id)
)

# 新方案（直接查询数据库）
result = self.session.execute(
    text("SELECT COUNT(*) FROM candidates WHERE candidate_id = :cid"),
    {"cid": candidate_id}
)
count = result.scalar()
```

**效果**：
- ✅ 绕过ORM缓存，直接查询数据库当前状态
- ✅ 确保验证结果与实际数据库一致
- ✅ 提高匹配保存成功率

# V3.10 版本更新说明

发布日期：2026-06-23

## 核心修复

### 自动修复数据不一致问题

**问题**：运行机器上可能出现ORM缓存与数据库实际数据不一致的情况，导致：
- 匹配过程能够读取到候选人数据
- 但保存时全部失败（FOREIGN KEY constraint failed）
- 例如：ORM返回499个候选人，但数据库实际只有66个

**根本原因**：
- SQLAlchemy session可能缓存了旧数据
- 数据库文件被替换/回滚后，应用没有重启
- 并发操作导致的数据不一致

**解决方案**：
在匹配开始前自动进行数据验证和修复：
1. 直接查询数据库获取实际存在的候选人ID集合
2. 过滤掉ORM返回但数据库中不存在的候选人
3. 只使用经过验证的有效候选人进行匹配
4. 在调试日志中记录数据验证和过滤过程

**效果**：
- ✅ 自动适应运行环境的实际数据状态
- ✅ 避免因数据不一致导致的批量保存失败
- ✅ 提高匹配成功率
- ✅ 减少无效的API调用

**调试日志增强**：
- 新增 `data_validation` 阶段：记录数据库实际候选人数量
- 新增 `data_filter` 阶段：记录过滤前后的数量对比
- 新增 `empty_after_filter` 阶段：记录过滤后无有效候选人的情况

# V3.08 版本更新说明

发布日期：2026-06-23

## 新增功能

### 智能匹配调试日志

为了更好地排查远程安装环境中的匹配失败问题，新增了独立的调试日志功能。

**功能特点：**

1. **自动生成** - 每次智能匹配时自动生成独立的JSON日志文件
2. **详细记录** - 记录整个匹配过程的所有关键信息：
   - 候选人数据验证（数量、ID范围、示例）
   - AI API交互详情（请求/响应、返回的ID列表）
   - 保存过程追踪（成功/失败数量、失败ID列表）
   - 错误详情（类型、信息）
   - 性能数据（耗时、超时情况）

3. **隐私保护** - 不包含敏感信息（简历内容、联系方式等）

4. **易于分享** - JSON格式，可直接发送给技术支持进行远程排查

**使用说明：**

- 日志位置：`logs/match_debug/` 目录
- 文件命名：`match_{岗位ID}_{时间戳}.json`
- 匹配失败时页面会提示日志位置
- 详细使用说明见 `docs/调试日志使用说明.md`

## 改进优化

### 错误提示增强

- 匹配失败时明确提示调试日志位置
- 区分自动匹配和手动匹配的提示信息
- 失败ID记录限制在前20个，避免日志过大

### 数据验证增强

- 保存匹配结果前验证候选人ID是否存在
- 提供更清晰的错误信息（从"FOREIGN KEY constraint failed"到"候选人 ID xxx 不存在"）

## 技术改进

- 新增 `MatchDebugLogger` 类（`recruitment_assistant/utils/match_debug_logger.py`）
- `match_candidates()` 方法支持传入调试日志记录器
- `_run_single_position_match()` 集成完整的调试日志记录流程
