# 三平台稳定性测试支撑（智联 / BOSS / 前程无忧）

**版本**：V2.51 计划
**日期**：2026-05-22
**作者**：sptwalker + Claude
**目标读者**：自己（HR 工具运维者）

## 1. 背景与目标

三平台采集（智联 / BOSS / 前程无忧）已经走通端到端流程，下一阶段是稳定性回归测试。每个平台需要多轮重复运行同一组候选人样本，定位以下问题：

- 应该自动点击的按钮没点上
- 简历预览弹窗未识别 / 识别错对象
- 本应自动完成、却错误进入"人工学习模式"
- 流程在某个候选人卡停几分钟

**本期不改采集业务逻辑**，只在现有日志/桥接里加 4 件事：

1. **看门狗超时**：候选人级 120s + 全局级 300s，超时自动跳过 / 终止并记录现场
2. **学习模式误进检测**：A2 + A3 两类（详见 §4）实时报警
3. **跨轮历史回归检测**：上一轮已成功下载的候选人，本轮又被弹学习模式 → 回归
4. **测试摘要自动生成**：每轮 `collect_finished` 时自动产出一份 markdown 报告

## 2. 范围

**改动**：

- `chrome_extension/content.js`：+ 响应 bridge 发来的 `skip_current_candidate` 控制消息
- `chrome_extension/manifest.json`：bump
- `recruitment_assistant/services/{boss,zhilian,qiancheng}_ws_bridge.py`：watchdog + misroute 判定 + 自动调 analyze 脚本
- `scripts/analyze_test_run.py`（新建）：JSONL → markdown 摘要 + history.json 维护
- `logs/test_runs/_persist_history.json`（运行时生成）：跨轮成功记录

**不动**：

- 业务流程、UI 页面、数据库、AI 服务

## 3. 看门狗

### 3.1 候选人级（120 秒）

**触发条件**：自 `candidate_clicked(candidate_id=X)` 起，bridge 维护该 cid 的 `last_event_at`，每次收到 `candidate_id == X` 的 emit 时刷新；若 `now - last_event_at >= 120s` 且尚未收到 X 的终态事件（`resume_persist_confirmed` / `candidate_skipped` / `resume_request_success`），触发超时。

**实现位置**：bridge 的 `handle_event` 协程附挂的 `asyncio.create_task` 看门狗。每次 emit 来到时刷新 `candidate_state[cid]["last_event_at"] = now`。

**触发动作**（按顺序）：

1. emit `watchdog_candidate_timeout` 到 JSONL，payload 含：`candidate_id / candidate_signature / last_event / elapsed_seconds`
2. UI 日志高亮一条："⚠️ 候选人 {sig} 看门狗超时，强制跳过"
3. 给 content.js 发 WebSocket 控制消息 `{type: "control", action: "skip_current_candidate", reason: "watchdog"}`
4. content.js 收到后调用 `skipCandidate(candidateId, signature, "watchdog_timeout")` + `proceedNextCandidate()`

### 3.2 全局级（300 秒）

**触发条件**：bridge 持续 300 秒未收到任何来自 content.js 的 emit（不含心跳 `heartbeat`）。

**触发动作**：

1. emit `watchdog_global_idle_timeout`
2. 调用 `_finish_crawl_task("failed", error_message="global_idle_timeout")`
3. UI 日志红色：「⚠️ 全局 5 分钟无事件，已强制终止本轮采集」

### 3.3 重置与开关

- 每个候选人的 watchdog 在 `candidate_clicked` 时启动 / 重置
- 候选人到达终态事件后停止该 cid 的看门狗
- 全局 watchdog 在 `collect_started` 启动，`collect_finished` 停止

## 4. 学习模式误进检测（misroute）

### 4.1 候选人级 mini state

每个 cid 维护：

```python
candidate_state[cid] = {
    "persisted": False,       # resume_persist_confirmed → True
    "auto_used": False,       # auto_download_click_used → True
    "learned_used": False,    # learned_download_click_used → True
    "last_event_at": now,
}
```

### 4.2 判定 A2：已落库又弹学习

收到 `manual_download_learning_required(candidate_id=X)` 时，若 `candidate_state[X]["persisted"] == True`：

→ emit `learning_misroute_detected`，payload：

```json
{
  "kind": "A2",
  "candidate_id": "X",
  "candidate_signature": "...",
  "note": "已落库却又弹学习模式（重复触发）"
}
```

→ UI 红色高亮日志一条

### 4.3 判定 A3：learnedClick 已用又弹学习

收到 `manual_download_learning_required(candidate_id=X)` 时，若 `candidate_state[X]["learned_used"] == True`：

→ emit `learning_misroute_detected(kind="A3", note="learnedClick 已用却又弹学习模式（学习成果未生效）")`

→ UI 红色高亮

### 4.4 不在范围内

- A1（自动策略全失败 → 进学习）：合理路径，不报
- A4（preview 未识别即进学习）：观察期，不报

## 5. 历史回归检测

### 5.1 历史载体

`logs/test_runs/_persist_history.json`，跨日累积，每平台一个 namespace：

```json
{
  "boss": {
    "袁先生在上海|28岁|本科": {
      "first_success_round": 3,
      "first_success_at": "2026-05-22T21:30:11",
      "last_success_round": 5,
      "last_success_at": "2026-05-22T22:14:08",
      "total_success": 4
    }
  },
  "zhilian": {...},
  "qiancheng": {...}
}
```

### 5.2 输入信号

- 加入历史：`resume_persist_confirmed.payload.candidate_signature`（三平台事件名一致，已确认）
- 检测回归：`manual_download_learning_required.payload.candidate_signature`

### 5.3 判定

每轮分析时：本轮 `manual_download_learning_required` 的 candidate_signature ∩ history\[platform\] 非空 → 列入摘要的「⚠️ 历史回归」section。

注：candidate_signature 由 content.js 内 `stripActivityText` 处理后产生，目前规则稳定。若日后 signature 规则变化，分析脚本检测到大量 history 中存在但本轮签名失配时给出警告"signature 规则可能改了"。

## 6. 测试摘要脚本

### 6.1 路径与用法

`scripts/analyze_test_run.py <jsonl_path>`

bridge 在 `collect_finished` 时自动调用：

```python
subprocess.Popen([sys.executable, "scripts/analyze_test_run.py", str(log_file)])
```

不阻塞主流程。

### 6.2 输出

`logs/test_runs/round_<round>_<platform>_<YYYYMMDD_HHMMSS>.md`

round 编号：扫描 `_persist_history.json` 旁的 `_round_counter.json`（每平台独立计数器），自动 +1。

### 6.3 摘要结构（A 模式）

### 6.4 history.json 维护

每次跑 analyze 时：

1. 读本轮 JSONL，把所有 `resume_persist_confirmed.candidate_signature` 加入 history\[platform\]
2. 已存在则更新 `last_success_round / last_success_at`，`total_success += 1`
3. 检测本轮 `manual_download_learning_required` 与 history 的交集 → 写进摘要 + UI 提示

## 9. 验收

测试启动后能看到：

1. 三平台启动时不再报"扩展版本不匹配"
2. 故意让某候选人卡死 120s，看门狗主动跳过并产生 `watchdog_candidate_timeout` 事件
3. 模拟 misroute（手动连续触发 learnedClick 又触发 learning_required），UI 红色高亮 `learning_misroute_detected`
4. 跑完一轮后 `logs/test_runs/round_X_平台_时间.md` 自动生成，含上述 7 个 section
5. 第二轮跑同一候选人，若不再触发学习则正常；若又进学习则在「历史回归」section 列出

## 10. 风险

| 风险 | 处置 |
|---|---|
| watchdog 误杀（页面慢但还活着）| 120s 已留余量；如仍误杀，下调到 90s 或上调到 180s 单点改一行常量 |
| content.js 收不到控制消息 | bridge 端有 `last_send_ack`；丢失时重发 1 次，仍失败则全局 watchdog 接管 |
| analyze 脚本异常导致摘要丢失 | bridge 用 `subprocess.Popen` 异步调用，异常不影响主流程；JSONL 原始日志保留 |
| signature 规则变化 → 历史失配 | analyze 检测到本轮签名规则与历史大面积失配时打印警告，由人工决定是否清空 history |
| 三平台事件名细微不一致 | 已确认 `resume_persist_confirmed` 三平台均使用；其他事件以 zhilian 为锚，差异由 platform-specific 字典在 analyze 里映射 |
