# 简历智采助手 — 多人生产级 ATS 架构升级规划

> 版本：v1（配合 A 组加固完成后编写）
> 定位转变：**单人桌面工具 → 多租户生产级 ATS**
> 本文是路线图与决策依据，不是实现细节说明书。每阶段给出「目标 / 关键改动 / 验收 / 风险」。

---

## 1. 现状定性

当前系统是一个**打磨用心的单人桌面工具**：

- **强**（招聘漏斗顶部）：三平台简历自动采集（Chrome 扩展，最难、最有壁垒）、AI 深度结构化解析（9 表 ATS 级数据）、AI 四维岗位匹配、STAR 面试大纲生成。
- **弱**（漏斗底部 + 团队协作）：无候选人状态流水线、无 Offer 管理、无对外通知、无报表分析、无登录/权限/多用户。流程止步于"面试结论=通过"。

### 四道架构级"坎"（不是补丁能过的）

| # | 坎 | 现状证据 | 生产级后果 |
|---|---|---|---|
| 1 | **单写 SQLite** | `resume_db.py` 默认 SQLite、未开 WAL；匹配 UI 全量 `page_size=10000` 加载 | 并发写锁 `database is locked`；低几千候选人即发卡 |
| 2 | **进程级单例共享可变状态** | `_failover_notices`（模块级 list）、`bridges.py` 的 `@st.cache_resource` WS 单例、`ws_server` 单连接槽 | 两个用户/标签页互相串台、互相踢连接 |
| 3 | **跨库外键无法约束** | `position_matches/interview_*` 在 SQLite 存 PG 的 `job_position.id`，无 FK；`delete_position` 软删只动 PG | 孤儿数据；`save_position_match` 已被迫降级用裸 sqlite3 绕 FK |
| 4 | **只进不退的迁移** | 运行时跑 `create_all`+手写 `ALTER ADD COLUMN`；alembic 存在但**运行时 0 调用** | 降级安装未定义；升级已破坏过一次（git `dc2869c`） |

**结论**：以当前骨架**加**多用户是错误路径。正确路径是——**以当前版本作为"已验证的业务原型"，用它跑通的业务逻辑驱动一次干净重写**。

---

## 2. 目标形态

多租户 Web ATS：

- **认证 + RBAC + 组织/团队**：招聘专员 / 用人经理 / 管理员角色，行级数据隔离。
- **完整候选人生命周期**：applied → 筛选 → 面试(多轮) → offer → 入职 / 拒绝，显式状态机。
- **对外闭环**：邮件/短信邀约、面试排期与提醒。
- **运营能力**：人才库标签、重复合并、报表分析（time-to-hire / 渠道转化 / 漏斗）。
- **可运维**：真实迁移框架、备份恢复、监控告警、CI/CD、灰度升级。

---

## 3. 分阶段路线

### 阶段 0：地基整改（不改业务功能，消灭四道坎）
**目标**：在不动业务的前提下，把架构改造到"可以安全承载多用户"的地基。

**关键改动**：
1. **统一单库**：把 SQLite 的 resume 全表迁入 PostgreSQL（或反之，但 PG 更适合多用户）。`JobPosition` 与候选人/匹配同库 → 真 FK 约束回归，`save_position_match` 去掉裸 sqlite3 绕行。
2. **alembic 成为唯一迁移源**：删除 `db.py`/`resume_db.py` 里的手写 `ALTER` + `create_all` 运行时迁移；启动改为 `alembic upgrade head`；补全从当前 schema 到统一 schema 的迁移脚本，支持回滚。
3. **消灭跨库孤儿**：`delete_position` 级联清理（或软删 + 过滤）匹配/邀约/大纲。
4. **去进程级单例**：`_failover_notices` 改为会话/请求级（`ContextVar` 或随请求传递）；WS 桥与 AI 服务不再 `@st.cache_resource` 全局单例。
5. **删除死代码**：PG 里未用的平行 `Candidate/Resume/WorkExperience` 模型 + `CandidateService`（下一个工程师的陷阱）。

**验收**：全量业务回归通过；两个并发会话互不串台；`alembic downgrade` 可回退一版。
**风险**：数据迁移（SQLite→PG）需一次性脚本 + 校验；停机窗口。

### 阶段 1：认证与多租户
**目标**：从"单用户"变"多用户安全隔离"。
**关键改动**：用户/角色/组织模型；登录（会话或 JWT）；所有查询加租户/团队维度的行级过滤；启用审计日志（`OperationAuditLog` 已有骨架，填 `operator/created_by`）。
**验收**：A 用户看不到 B 用户/别组的候选人；越权访问被拒；关键写操作有审计记录。
**风险**：每个查询都要带隔离维度，遗漏即数据泄露——需在 ORM/仓储层统一强制，不靠调用方自觉。

### 阶段 2：业务闭环补全
**目标**：从"采集+初筛加速器"变"完整 ATS"。
**关键改动**：
- **候选人状态机**：显式枚举 + 合法流转校验（applied→筛选→面试→offer→入职/拒绝/暂缓）。
- **Offer 管理**：Offer 模型（薪资/职级/入职日/审批状态）+ 审批流。
- **对外通知集成**：邮件（SMTP/服务商 API）+ 短信；邀约/排期/提醒模板。
- **人才库标签 + 重复合并 UI**（当前只"拦截"重复，不"合并"）。
**验收**：一个候选人可从投递走到入职并全程留痕；邀约真能发到候选人；两条重复记录可合并为一。
**风险**：通知涉及外部服务与合规（见阶段 4）；状态机改动触及大量现有 UI。

### 阶段 3：规模化与运营
**目标**：从"几百人"到"几万人"稳定运行。
**关键改动**：真实分页/服务端检索（去掉 `page_size=10000` 全量加载）；报表分析看板；备份/恢复自动化（pg_dump 定时 + 一键恢复）；监控告警（健康检查、错误率、LLM 用量/成本）；CI/CD + 灰度/回滚发布。
**验收**：万级候选人下列表/匹配/检索响应可接受；备份可一键恢复；发布可回滚。
**风险**：LLM 匹配 O(N·M) 的成本与限流需要队列化/批处理与预算控制。

### 阶段 4：合规与安全（贯穿，红线）
**目标**：满足处理招聘 PII 的法律与安全要求。
**关键改动**：PII 静态加密（库级或字段级，电话/邮箱/微信）+ 访问控制 + 脱敏展示；密钥集中管理（密钥管理服务，杜绝明文入库/入包）；**采集 ToS / PIPL 合规评估**（扩展式抓取三平台附件 + 无第一方关系存 PII 是真实法律敞口——产品/法务决策项，非工程能独立解决）。
**验收**：静态数据加密、最小权限、可审计；有合规结论与数据处理协议。
**风险**：采集合规可能要求改变产品形态（如改用平台官方 API / 候选人授权）。

## 4. 技术选型建议

### 4.1 前后端形态：Streamlit 的取舍
- **Streamlit 的根本限制**：全脚本 rerun 模型 + 进程级状态，天然是单用户内部工具形态；多用户下的会话隔离、并发、认证都要逆着框架做。
- **建议**：生产级 ATS 走 **前后端分离**——后端 FastAPI（复用现有 service 层业务逻辑，这是当前项目最有价值的沉淀）+ 前端独立框架（React/Vue）。**Streamlit 可保留**为内部运营/调试工具。
- **迁移友好点**：现有 `recruitment_assistant/services/*` 与 `schemas/*` 是与 UI 解耦的，**可直接被 FastAPI 复用**——这是"干净重写"成本可控的关键。

### 4.2 部署形态
- 桌面版继续用 Inno Setup 单机分发（内部/离线场景）。
- 生产版容器化（Docker）：应用 + PostgreSQL + 反向代理；单实例起步，无状态化后可多实例横向扩展。
- 采集：扩展→WS→后端的模型在多用户下需重新设计（每用户一个采集会话/连接，不能全局单例）。

---

## 5. 迁移策略

1. **不在现骨架上加多用户**——用当前版本验证过的业务规则（解析字段、四维匹配、去重逻辑、面试流程）作为需求真源，驱动阶段 0 的干净重写。
2. **数据迁移脚本**：SQLite `resume_archive.db` → 统一 PG schema，含校验（行数、抽样比对）与回滚点。
3. **并行运行期**：新系统上线前，老桌面版继续可用；提供导出（已在 A 组交付 Excel/DB 备份）作为过渡桥梁。

## 7. 里程碑建议（粗粒度）

- **M0（已完成，A 组）**：备份/导出、核心测试+CI、迁移失败可见、密钥移出 git。
- **M1（阶段 0）**：单库化 + alembic + 去单例 + 删死代码 —— 地基就绪。
- **M2（阶段 1）**：认证 + 多租户隔离 + 审计。
- **M3（阶段 2）**：状态机 + Offer + 通知 + 标签/合并 —— 成为完整 ATS。
- **M4（阶段 3/4）**：规模化、报表、运维、PII 加密与合规 —— 生产级。

> 每个里程碑独立可交付、可回退。M1 是分水岭：完成前不对外承诺"多用户"。
