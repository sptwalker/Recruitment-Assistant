# 智能匹配调试日志使用说明

## 功能说明

从 V3.07 版本开始，系统在执行智能匹配时会自动生成详细的调试日志文件，用于远程排查匹配失败问题。

## 日志位置

调试日志保存在：`logs/match_debug/` 目录下

文件命名格式：`match_{岗位ID}_{时间戳}.json`

例如：`match_22_20260623_195536.json`

## 日志内容

每个调试日志文件包含以下信息：

### 基本信息
- `position_id`: 岗位ID
- `position_title`: 岗位标题
- `start_time`: 匹配开始时间
- `end_time`: 匹配结束时间
- `duration_seconds`: 总耗时（秒）

### 详细日志条目 (log_entries)

每条日志包含：
- `timestamp`: 时间戳
- `stage`: 阶段标识
- `message`: 描述信息
- `data`: 详细数据（可选）

### 阶段说明

1. **init** - 初始化
   - 记录岗位信息和开始时间

2. **candidates** - 候选人数据读取
   - `total_count`: ORM查询返回的候选人总数
   - `dict_count`: 传递给AI的候选人摘要数量
   - `id_range`: 候选人ID范围（最小值、最大值）
   - `first_10_ids`: 前10个候选人ID
   - `last_10_ids`: 最后10个候选人ID
   - `sample_candidate`: 第一个候选人示例

3. **ai_request** - AI API请求
   - `batch_index`: 批次编号
   - `candidate_count`: 本批次候选人数量
   - `jd_preview`: 岗位要求预览（前200字符）

4. **ai_response** - AI API响应
   - `batch_index`: 批次编号
   - `result_count`: AI返回的匹配结果数量
   - `returned_ids`: AI返回的所有候选人ID列表
   - `sample_result`: 第一条匹配结果示例

5. **save_attempt** - 开始保存
   - `total_results`: 待保存的匹配结果总数
   - `id_range`: 候选人ID范围
   - `first_10_ids`: 前10个ID
   - `last_10_ids`: 最后10个ID

6. **save_result** - 保存完成
   - `success_count`: 成功保存的数量
   - `fail_count`: 失败的数量
   - `failed_ids`: 失败的候选人ID列表（最多记录前20个）

7. **error** - 错误记录
   - `error_type`: 错误类型
   - `error_message`: 错误信息

8. **timeout** - 超时记录
   - 记录超时时已完成的批次数

## 使用场景

### 场景1：所有匹配结果都保存失败

**症状**：智能匹配后显示"暂无匹配结果"，或提示保存失败

**排查步骤**：

1. 打开最新的 `match_xxx.json` 文件
2. 检查 `candidates` 阶段：
   - `total_count` 和 `dict_count` 是否一致？
   - `id_range` 范围是否合理？
3. 检查 `ai_response` 阶段：
   - `returned_ids` 是否包含有效的候选人ID？
4. 检查 `save_result` 阶段：
   - `fail_count` 是否等于总数？
   - `failed_ids` 列表中的ID是否存在于 `candidates` 阶段的ID列表中？

**常见问题**：
- 如果 `returned_ids` 中的ID不在 `candidates.first_10_ids` 或 `last_10_ids` 范围内，说明AI返回了不存在的候选人ID
- 如果 `save_result.failed_ids` 都在合理范围内，可能是数据库外键约束问题

### 场景2：部分匹配结果保存失败

**排查步骤**：

1. 对比 `ai_response.returned_ids` 和 `save_result.failed_ids`
2. 检查失败的ID是否有共同特征（如连续区间、特定范围）

### 场景3：匹配超时

**排查步骤**：

1. 检查 `timeout` 阶段记录
2. 查看 `duration_seconds` 总耗时
3. 检查已完成的批次数

## 发送日志给技术支持

如果遇到匹配失败问题需要远程排查：

1. 找到 `logs/match_debug/` 目录下最新的日志文件
2. 将该 JSON 文件发送给技术支持
3. 日志文件不包含敏感信息（简历详细内容、联系方式等），只包含：
   - 岗位ID和标题
   - 候选人ID列表和数量统计
   - 匹配评分和简短评语
   - 错误信息

## 注意事项

1. 每次匹配都会生成新的日志文件，不会覆盖旧文件
2. 建议定期清理旧的调试日志（保留最近1-2周即可）
3. 日志文件为JSON格式，可用任何文本编辑器打开查看
4. 单个日志文件大小通常在几KB到几十KB之间
